# Deferred Issue: SKOS Vocabulary Formalization

## Status: Deferred

## Summary

The system uses three tiers of controlled vocabulary (see ARCHITECTURE.md,
"Vocabulary Tiers"). Currently these tiers are implicit — the structural
vocabulary is encoded as OWL classes, the type vocabulary as OWL subclasses
generated by SchemaBuilder, and the value vocabulary as string literals
constrained by SHACL `sh:in`.

SKOS (Simple Knowledge Organization System) is the natural W3C standard for
making these vocabulary tiers explicit and machine-readable in RDF.

## What SKOS Would Provide

- **`skos:ConceptScheme`** for each vocabulary tier or scope (e.g., a
  `kc:StructuralVocabulary` scheme, an `mtg:TypeVocabulary` scheme, an
  `mtg:DispositionVocabulary` scheme).
- **`skos:Concept`** for each term, with `skos:inScheme` linking it to its
  scheme and `skos:prefLabel` for human-readable names.
- **`skos:broader` / `skos:narrower`** for vocabulary extension across layers
  (e.g., `mtg:Color` is narrower than `kc:Vertex` in the type vocabulary).
- **Queryability**: SPARQL queries over concept hierarchies (e.g., "which
  model families extend the core vertex vocabulary?").
- **Self-documentation**: the vocabulary is a first-class RDF resource, not
  buried inside SHACL constraints or OWL class hierarchies.

## Trade-offs

**Benefits:**
- Extensibility: a second model family could extend an existing value
  vocabulary (add values) rather than replacing it.
- The layer boundary becomes visible in RDF, not just in Python code and
  documentation.
- Standard W3C practice for controlled vocabularies.

**Costs:**
- Adds another W3C standard dependency (SKOS) to an already multi-standard
  stack (OWL + SHACL + SPARQL).
- Attribute values would become IRIs (e.g., `mtg:adjacent`) instead of string
  literals (`"adjacent"`). This changes the Python API surface.
- For a single-model proof-of-concept, the added complexity may not pay off.

## Recommendation

Defer until the framework needs to support multiple model families or
vocabulary governance across teams. The current `vocab()` → `sh:in` mechanism
is sufficient for the MTG demo and WP3 implementation.

## Affected Components

- `kc/resources/kc_core.ttl` — would add `skos:` prefix and structural concept scheme
- `kc/schema.py` — `vocab()` would generate `skos:ConceptScheme` + `skos:Concept`
- `kc/graph.py` — attribute values would be IRIs instead of strings
- All SPARQL queries referencing attribute values
- All test fixtures using string attribute values
